# STRUCTURE VECTORS
# Structure vectors are defined below in the following format:
#
# SV type (Rho or FFG)
#     name: string; must correspond to string name used in tree definition
#     neighbors: list of neighboring elements
#     components: a list of arbitrary names of sub-splines for generating SV
#     num_knots: number of knots for each sub-spline
#
# Notes:
#     - an SV with multiple sub-splines is generated by taking the outer
#     products of each sub-spline in the order listed in the 'components' field.
#     - For the FFG SVs, the settings for the g spline should always be listed
#     last in num_knots.

Rho
    name: rho_0
    neighbors: Mo
    cutoffs: 2.4 5.2
    num_knots: 7

Rho
    name: rho_1
    neighbors: Ti
    cutoffs: 2.4 5.2
    num_knots: 7

FFG
    name: ffg_0_1
    neighbors: Mo Ti
    cutoffs: 2.4 5.2
    num_knots: 7 7 7

FFG
    name: ffg_1_1
    neighbors: Ti Ti
    cutoffs: 2.4 5.2
    num_knots: 7 7

################################################################################
# TREE
# A string representation of the equation tree is given.
# The two tree definitions below are equivalent. Either format can be used, but
# the first version with angle brackets, parentheses, and commas, might be more
# human-readable. Sub-trees for different elements must be separated by '|'.
# Currently only the 'add' function is supported.
# Structure vectors names must start with one of the names defined in this file.
# Note: duplicate SVs have independent parameters.

Tree
<Mo> add(rho_0, add(ffg_0_1, ffg_1_1)) | <Ti> add(rho_1, rho_1)
# Mo add rho_0 add ffg_0_1 ffg_1_1 | Ti add rho_1 rho_1

################################################################################
# PARAMETERS 
# This section must come after the TREE section.
# One parameter per line. Ordered according to the structure vectors defined
# in the tree. In this example, the parameters of rho_0 come first, ffg_0_1
# second, ffg_1_1 third, the first rho_1 fourth, then the second rho_1 fifth.
# When SVs have multiple components, the ordering of their parameters must match
# the order that their source code assumes. In the case of FFG SVs, the assumed
# order is f_0, f_1, g (e.g. ffg_0_1) or f_1, g (e.g. ffg_1_1). So to be
# explicit, the parameter ordering here is [rho_A, f_A, f_B, g_A_B, f_A, g_A_A,
# rho_B, rho_B]. Parameters for a single sub-spline are ordered as [knot_1, ...,
# knot_N, d0, dN].
